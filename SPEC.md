# Wraith

Stealth addresses for Horizen. Send and receive any asset privately with no on-chain link between sender and recipient.

---

## What it is

Wraith is two things shipped together: an SDK that any Horizen developer can drop into their dApp to add stealth address support, and a web app that uses the SDK to give regular users a clean interface for sending and receiving privately.

The SDK is the core contribution. The web app is the demonstration.

Both are built on ERC-5564 (Stealth Address Messenger) and ERC-6538 (Stealth Meta-Address Registry), using the secp256k1 cryptographic scheme.

---

## What you can send

Stealth addresses are asset-agnostic. The generated stealth address is a regular wallet address on Horizen and can hold anything that address is capable of: the native gas token, ERC-20 tokens, NFTs, or any other asset on the chain. The Messenger contract only handles the announcement. The actual asset transfer is a separate transaction directed at the stealth address. Wraith works for any asset, not any specific one.

---

## How stealth addresses work

A stealth address is a regular wallet address generated by the sender on behalf of the recipient, without any coordination between them. To outside observers it looks like a transaction to a random fresh account. Only the intended recipient can recognise it as theirs and spend from it.

Every user who wants to receive stealth transfers has a stealth meta-address. This is a publicly shareable string that encodes two public keys: a spending public key and a viewing public key. Sharing it allows anyone to send to you privately without any coordination.

When someone sends to you, they take your stealth meta-address, generate a random ephemeral key, and use elliptic curve math to derive a one-time stealth address that only you can control. They send assets to that address and publish a small announcement on-chain containing just enough information for you to find it, but nothing useful to anyone else.

To find incoming transfers, you scan on-chain announcements using your viewing private key. Each announcement contains a view tag — a single byte that lets you skip irrelevant announcements quickly, cutting the workload by around 255 out of every 256. On a match, you use your spending private key to derive the private key controlling the stealth address. At that point the stealth address is just a regular wallet you own.

The viewing key and spending key serve different roles. The viewing key finds your stealth addresses. The spending key accesses them. This means you can hand a viewing key to a trusted scanning service without risking funds.

---

## Key derivation

Users never handle raw private keys. Wraith derives the spending and viewing keys from a wallet signature using the RFC 6979 approach for deterministic key generation.

When a user connects their wallet, Wraith asks them to sign a fixed, human-readable message specific to Wraith and the Horizen network. The signature bytes are sliced and hashed to produce the spending key and viewing key. Because RFC 6979 makes the same wallet produce the same signature for the same message every time, the derived keys are always recoverable. Reconnect your wallet later, sign again, and the same keys come back.

Keys only exist in memory during the session. Wraith never writes them to localStorage, cookies, or any persistent store. When the session ends, they are gone.

## Human-readable names

Wraith names are short identifiers that map to a stealth meta-address. Instead of sharing a long hex string, a user can share something like `truth.wraith`.

The name-to-wallet mapping is on-chain in a Wraith-specific naming registry. The wallet-to-meta-address mapping is the standard ERC-6538 registry. Together they form a two-hop lookup: name → wallet → meta-address.

Names are unique, lowercase, alphanumeric, and between 3 and 32 characters. Once registered, a name belongs to the registrant permanently — there is no expiry. The owner can update the wallet it points to or release it.

The privacy consideration here is registration. The wallet used to register a Wraith name is linked to that name on-chain. If that wallet is already connected to a public identity, the link is established. Wraith warns users about this during setup and recommends registering from a fresh wallet. When the relayer is live, it can handle the registration transaction so the registrant wallet is never directly visible.

## UX privacy protections

Beyond the cryptographic guarantees, Wraith actively protects users from the behavioural mistakes that cause real deanonymisation in practice. Research on Umbra, the leading stealth address implementation on Ethereum, found 48.5% of transactions were deanonymised through user behaviour rather than cryptographic breaks.

Wraith addresses this directly:

- **Connected wallet blocking.** The app blocks withdrawals to any wallet address that was connected during the same session. Sending funds back to the wallet that sent them — or to the wallet used during setup — is the most common deanonymisation vector.

- **Fresh destination enforcement.** Every spend from a stealth address requires a destination address that has not appeared in any prior stealth transaction in the session. Reusing addresses across stealth withdrawals creates linkable patterns.

- **Timing suggestions.** The app suggests a waiting period before spending from a newly funded stealth address. Immediate withdrawals after deposits create temporal correlations that chain analysis tools exploit.

- **Balance batching.** When a user has received multiple stealth transfers, the app supports batching them into a single outbound transaction to a fresh address. This reduces the on-chain footprint from many small withdrawals to one.

- **Plain language warnings.** At every step where a user's action could reduce their privacy, the app explains what is and is not private in plain language. No jargon, no assumptions about prior knowledge.

## Withdrawals and batching

The hardest part of stealth addresses is spending from them. The recipient has to import a raw private key into their wallet, fund it with gas, and manually transfer assets out. Wraith removes all of this friction.

When a user scans and finds incoming stealth transfers, the app shows each stealth address with its balance. To withdraw, the user enters a fresh destination address and clicks once. Behind the scenes, Wraith derives the stealth private key in memory, creates and signs the withdrawal transaction, broadcasts it, and discards the key. The private key exists for the duration of the signing operation and nowhere else. It is never written to disk, never displayed to the user, and never leaves the browser. The trust model is the open-source codebase itself — the code is auditable and the app is fully client-side.

For users with multiple stealth addresses holding funds, Wraith supports consolidated withdrawals. The user selects which stealth addresses to withdraw from, enters one destination, and the app executes each withdrawal in sequence — deriving, signing, and discarding each key one at a time. The result is a single destination receiving funds from multiple stealth sources, with no private keys retained anywhere.

On the sending side, batch sending lets a user send to multiple recipients in one session. Each recipient gets their own freshly generated stealth address and announcement. The sender confirms each transfer individually in their wallet but doesn't have to navigate back and forth — the app queues them and walks through the flow.

## Scanning via Goldsky

The Messenger contract emits an Announcement event every time a stealth transfer is made. As the chain grows, scanning raw RPC logs from genesis becomes increasingly slow.

Wraith uses Goldsky — Horizen's official indexing partner — to index every Announcement event in real time. When a user opens the receive page, Wraith queries Goldsky via GraphQL to fetch all announcements instantly, then runs the viewing key check locally against that list. The expensive chain scanning is replaced by a fast API call.

## The contracts

Three contracts are deployed on Horizen.

The **Messenger** contract is the announcement board. Every time someone sends a stealth transfer, they call this contract to publish the ephemeral public key and view tag. Recipients scan these announcements to find their incoming transfers. This follows the ERC-5564 standard.

The **Registry** contract is where users publish their stealth meta-address so others can look it up by their regular wallet address. A user registers once and anyone can retrieve their meta-address on-chain. This follows the ERC-6538 standard.

The **Names** contract is the Wraith naming registry. It maps human-readable names to wallet addresses. Combined with the Registry contract, it enables the full name → wallet → meta-address lookup chain.

---

## The SDK

The SDK is a TypeScript package published to npm as `@wraith/sdk`. It handles all the stealth address cryptography and is wallet-library agnostic — it works with viem, ethers, or anything that can produce a signature.

It covers: deriving stealth keys from a wallet signature, constructing the stealth meta-address, generating a one-time stealth address for a given recipient, scanning a list of announcements to find transfers belonging to a given viewing key, and deriving the private key needed to spend from a stealth address.

The SDK makes no RPC calls itself. It takes raw data in and returns results out. The developer controls the provider, wallet connection, and chain interaction. This keeps it composable and easy to test.

---

## The web app

The web app is a Vite + React application that uses the SDK and connects to Horizen mainnet, with a testnet toggle for developers.

It has four main sections.

**Setup** is where a user connects their wallet, signs the Wraith message to derive their keys, and registers their stealth meta-address on-chain. It also offers the option to choose a human-readable Wraith name that maps to their meta-address.

**Send** is where a user enters a recipient's stealth meta-address or Wraith name, selects an asset, enters an amount, and completes the transfer. The app resolves Wraith names automatically via the on-chain naming registry. Batch sending is supported — a user can send to multiple recipients in a single session, generating a separate stealth address for each.

**Receive** is where Goldsky-powered scanning surfaces all incoming stealth transfers. The user can view balances across all their stealth addresses and withdraw with one click. The app derives the stealth private key in memory, signs the withdrawal transaction, and immediately discards the key — no private key is ever stored, displayed, or exported. Users can consolidate multiple stealth address balances into a single withdrawal to a fresh destination, or withdraw individually. The app actively prevents common privacy mistakes — it warns against withdrawing to a connected wallet and enforces fresh destination addresses.

**About** explains how everything works and links to the SDK on npm and the GitHub repo.

The app supports MetaMask, Coinbase Wallet, WalletConnect, and any EIP-1193 compatible wallet.

---

## Monorepo structure

Wraith is a monorepo using pnpm workspaces with three parts: the SDK package, the web app, and a Hardhat project for the contracts. The SDK and web app live under a packages directory. The contracts sit at the root level alongside them.

## Roadmap

- [x] **Atomic send-and-announce** — WraithSender contract executes transfer + announcement in a single atomic transaction.
- [x] **Token detection on stealth addresses** — Receive page scans ETH + ERC-20 balances (ZEN, USDC) on each stealth address. Dust balances are hidden by default with a toggle to reveal.
- [x] **Gas-sponsored withdrawals via EIP-7702** — WraithWithdrawer contract deployed as EIP-7702 delegation target. Horizen supports EIP-7702, enabling gas sponsorship on plain EOA stealth addresses for both native tokens and ERC-20s.
- [x] **Batch send** — WraithSender supports batchSendETH and batchSendERC20 for sending to multiple stealth addresses in one transaction. UI has a Single/Batch toggle.
- [x] **Batch withdraw** — Receive page supports selecting multiple stealth addresses and withdrawing all to one destination sequentially.
- [x] **One-click withdrawal** — Derive stealth private key in memory, sign withdrawal, discard key. No key ever stored or displayed unless explicitly revealed.
- [x] **Subgraph indexing via Goldsky** — Announcements indexed in real time via Goldsky subgraph. Scanning queries GraphQL instead of raw RPC logs.
- [ ] **Human-readable names** — WraithNames contract for on-chain name-to-wallet mapping. Two-hop lookup: name → wallet → meta-address.
- [ ] **Payment links** — Shareable URLs like `wraith.app/pay/truth.wraith` that open the send page pre-filled with the recipient.
- [ ] **Wraith name search** — Public lookup page for searching Wraith names and retrieving meta-addresses.
- [ ] **UX privacy protections** — Connected wallet blocking, fresh destination enforcement, timing suggestions, balance batching warnings, plain language warnings at every step.
- [ ] **Multi-recipient airdrop contract** — On-chain batch that takes meta-addresses + amounts, generates stealth addresses, transfers, and announces all in one transaction.
- [ ] **Paymaster** — ERC-4337 paymaster as an alternative to EIP-7702, for smart contract wallet compatibility.
- [ ] **ZK spending trail** — ZK shielded pool for withdrawals that breaks the on-chain link between stealth address and spending destination.
- [ ] **FHE-DKSAP** — FHE-based Dual Key Stealth Address Protocol for quantum-resistant stealth addresses and trustless outsourced scanning without sharing the viewing key.
- [ ] **Mobile** — Mobile-optimised experience for scanning and spending.
