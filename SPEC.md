# Wraith

Stealth addresses for Horizen. Send and receive any asset privately with no on-chain link between sender and recipient.

---

## What it is

Wraith is two things shipped together: an SDK that any Horizen developer can use to add stealth address support to their own dApp, and a web app that uses the SDK to give regular users a clean interface for sending and receiving privately.

The SDK is the core contribution. The web app is the demonstration.

Both are built on ERC-5564 (Stealth Address Messenger) and ERC-6538 (Stealth Meta-Address Registry), using the secp256k1 cryptographic scheme.

---

## What you can send

Stealth addresses are asset-agnostic. The generated stealth address is just a regular wallet address on Horizen, so it can receive anything that address is capable of holding: the native gas token, ERC-20 tokens, NFTs, or any other asset on the chain. The Messenger contract only handles the announcement. The actual asset transfer is a completely separate transaction that happens to be directed at the stealth address. Wraith works for any asset, not any specific one.

---

## How stealth addresses work

A stealth address is a regular wallet address generated by the sender on behalf of the recipient, without any coordination between them. To outside observers it looks like a transaction to a random fresh account. Only the intended recipient can recognise it as theirs and spend from it.

Every user who wants to receive stealth transfers has a stealth meta-address. This is a publicly shareable string that encodes two public keys: a spending public key and a viewing public key. Sharing it allows anyone to send to you privately.

When someone sends to you, they take your stealth meta-address, generate a random ephemeral key, and use elliptic curve math to derive a one-time stealth address that only you can control. They send assets to that address and publish a small announcement on-chain containing enough information for you to find it, but nothing useful to anyone else.

To find incoming transfers, you scan all on-chain announcements using your viewing private key. Each announcement contains a view tag, a single byte that lets you skip irrelevant announcements quickly — only about one in 256 needs a full check. When you find a match, you use your spending private key to derive the private key that controls the stealth address. At that point the stealth address is just a regular wallet you own — import the key, spend from it, do whatever you want with it.

The viewing key and spending key serve different roles. The viewing key finds your stealth addresses. The spending key accesses them. This separation means you can hand a viewing key to a trusted service to do scanning on your behalf, without ever putting your funds at risk.

---

## Key derivation

Users never handle raw private keys. Wraith derives the spending and viewing keys from a wallet signature, using the RFC 6979 approach for deterministic key generation.

When a user connects their wallet, Wraith asks them to sign a fixed, human-readable message specific to Wraith and the Horizen network. The resulting signature bytes are sliced and hashed to produce the spending key and the viewing key. Because RFC 6979 makes the same wallet produce the same signature for the same message every time, the derived keys are always recoverable. If you lose access and reconnect later, signing again produces identical keys.

The keys only ever exist in memory during the session. Wraith never writes them to localStorage, cookies, or anywhere persistent. When the session ends, they are gone.

## The contracts

Two contracts are deployed on Horizen mainnet and testnet.

The Messenger contract is the announcement board. Every time someone sends a stealth transfer, they call this contract to publish the ephemeral public key and view tag. Recipients scan these announcements to find their incoming transfers. This follows the ERC-5564 standard.

The Registry contract is where users publish their stealth meta-address so others can look it up by their regular wallet address. Instead of sharing the meta-address manually every time, a user registers it once and anyone can retrieve it on-chain. This follows the ERC-6538 standard.

## The SDK

The SDK is a TypeScript package published to npm as `@wraith/sdk`. It handles all the stealth address cryptography and is designed to be wallet-library agnostic — it works with viem, ethers, or anything that can produce a signature.

It covers four things: deriving stealth keys from a wallet signature and constructing the stealth meta-address, generating a one-time stealth address for a given recipient, scanning a list of on-chain announcements to find transfers belonging to a given viewing key, and deriving the private key needed to spend from a stealth address.

The SDK makes no RPC calls itself. It takes raw data in and returns results out. The developer controls the provider, the wallet connection, and the chain interaction. This keeps it composable and easy to test.

---

## The web app

The web app is a Next.js application that uses the SDK and connects to Horizen mainnet, with a testnet toggle for developers.

It has four main sections. Setup is where a user first connects their wallet, signs the Wraith message to derive their keys, and registers their stealth meta-address on-chain. Send is where a user pastes a recipient's stealth meta-address, picks an asset, enters an amount, and sends privately in a single transaction. Receive is where a user scans all on-chain announcements to see incoming stealth transfers and spend from them. About explains how everything works and links to the SDK and the GitHub repo.

The app supports MetaMask, Coinbase Wallet, WalletConnect, and any EIP-1193 compatible wallet.

---

## Monorepo structure

Wraith is a monorepo using pnpm workspaces. It has three parts: the SDK package, the web app, and a Foundry project for the contracts. The SDK and web app are under a packages directory. The contracts sit at the root level alongside them.

## Roadmap

The v1 scope is the core: stealth sends, scanning, and spending. Everything after that is a v2 concern.

Gas abstraction is the most useful next step. When a stealth address receives only an ERC-20 token with no native gas token attached, the recipient needs to fund it separately before they can spend. A relayer that sponsors the first transaction out of a stealth address — reimbursed from the transferred amount — would remove this friction entirely and make the experience seamless for non-technical users.

Other things being considered for later: a parsing service that scans announcements server-side so users do not have to scan from genesis every time they open the app, ENS-style human-readable names for stealth meta-addresses, and multi-asset batch sends where a sender can transfer multiple tokens to a stealth address in a single transaction.
